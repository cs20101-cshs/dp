## 資訊安全技術概論
**某網站存在跨站請求偽造（Cross Site Request Forgery, CSRF）漏洞，攻擊者可能會利用該漏洞發起攻擊。下列何種方法最常用來防止CSRF攻擊？**  
(A) 在 URL 中加入 Session ID  
(B) 將所有表單設為唯讀權限（Read only）  
***(C) 在使用者的請求中加入 Token***  
(D) 將所有網頁設置為不可被搜索引擎索引  

關鍵判斷：  
CSRF 是攻擊者利用使用者已登入某網站的狀態，誘導瀏覽器向該網站送出非預期請求。  

常見防禦方式包含：  
- 加入 CSRF Token
- 檢查 Origin / Referer
- SameSite Cookie
- 重要操作要求重新驗證  

其中最常見、最標準的作法是使用 `CSRF Token`。

**PHP 程式語言中有好幾個可以用來執行作業系統指令的危險函數，使用的時候必須非常謹慎，如果程式撰寫不當，有機會被攻擊者拿來利用，進一步取得網頁伺服器的控制權。請問下列哪一個語法不是用來執行作業系統指令的PHP函數？**  
(A) system (B) passthru ***(C) eval*** (D) popen  

關鍵判斷：  
PHP 中常見可執行作業系統指令的危險函數包含：      
- `system()`
- `passthru()`
- `exec()`
- `shell_exec()`
- `popen()`
- 反引號運算子 `` `command` ``

`eval()` 則是執行 PHP 程式碼字串，不是直接執行 OS command。  

判斷規則：  
- `system / passthru / exec / shell_exec / popen` → OS command execution
- `eval` → PHP code execution

**軟體組成分析（Software Composition Analysis, SCA）主要是針對程式設計的哪一個部分進行分析？**  
(A) 所開發的原始碼是否有設計缺陷  
(B) 編譯後的執行檔是否有錯誤配置  
(C) 針對二進位檔進行 Security Hardening 檢查  
***(D) 軟體元件是否有已知漏洞***  

關鍵判斷：  
`SCA`（Software Composition Analysis，軟體組成分析）主要分析軟體中使用的第三方元件、開源套件與相依函式庫，確認是否存在：  
- 已知漏洞
- 高風險版本
- 授權問題
- 過期或不再維護的套件
- 相依性風險

判斷規則：  
- SCA = 看第三方套件 / 開源元件 / 相依性有沒有已知漏洞
- SAST = 看自己寫的原始碼弱點
- DAST = 跑起來從外部測

**下列哪一種密碼雜湊的強度最高、破解速度非常慢，即使用常見的密碼破解軟體 hashcat 來進行破解，也需要耗費非常久的時間才能破解出來？**  
(A) NTLM ***(B) bcrypt*** (C) MD5 (D) SHA1  

關鍵判斷：  
`bcrypt` 是專門用於密碼儲存的雜湊演算法，設計上會刻意增加計算成本，使暴力破解或字典攻擊變慢。

相較之下：  
- `MD5`：速度快，已不適合密碼儲存。
- `SHA1`：速度快，且已不建議用於安全用途。
- `NTLM`：Windows 常見密碼雜湊格式，破解速度快，容易被 hashcat 高速嘗試。
- `bcrypt`：慢速密碼雜湊，適合密碼儲存。

判斷規則：  
- 密碼雜湊強度：bcrypt > NTLM / MD5 / SHA1
- bcrypt = 慢，適合密碼
- MD5 / SHA1 / NTLM = 快，容易被爆破

**關於 Linux 日誌的相關描述，下列何者錯誤？**  
(A) `/var/log/boot.log` 記錄開機時系統核心偵測與啟動硬體，及各種核心支援的功能啟動等，只存放當次開機的資訊  
(B) `/var/log/cron` 記錄 crontab 排程有沒有實際被進行  
(C) `/var/log/dmesg` 記錄系統在開機的時候核心偵測過程所產生的各項資訊  
***(D) `/var/log/messages` 記錄系統上面所有的帳號最近一次登入系統時的相關資訊***  

關鍵判斷：  
`/var/log/messages` 通常記錄一般系統訊息，例如系統服務、核心、網路、daemon 等相關訊息，不是使用者最近一次登入紀錄。  

使用者最近一次登入資訊通常與下列兩者有關：  
- `/var/log/lastlog`
- `lastlog` 指令

判斷規則：  
- `/var/log/boot.log` → 開機服務訊息
- `/var/log/cron` → 排程 cron
- `/var/log/dmesg` → kernel 開機偵測訊息
- `/var/log/messages` → 一般系統訊息
- `/var/log/lastlog` → 使用者最後登入紀錄

**物聯網架構具有三個層次，包含感知層、網路層、應用層，分別應用於 Bluetooth、Wi-Fi、4G 的加密保護機制為下列何項？**  
***(A) LE Encryption；WPA/WPA2；EEA（EPS Encryption Algorithm）***  
(B) AES；LE Encryption；WEP  
(C) LE Encryption；WPA/WPA2；SNOW 3G  
(D) KASUMI；SNOW 3G；WPA-PSK/WPA2-PSK  

關鍵判斷：

## 資訊安全管理概論
**組織向第三方驗證公司申請ISO 27001資訊安全管理系統證書，下列步驟何項正確？**  
(A) 初次驗證通過後，需要不定期重新驗證以取得證書  
***(B) 初次申請驗證流程，分為文件審查與實地審查兩個階段***  
(C) 重新審查只需要進行文件審查即可  
(D) 組織取得通過驗證證書後，三年進行一次重新審查即可延續證書有效期限   

關鍵判斷：  
組織向第三方驗證公司申請 ISO 27001 ISMS 驗證時，初次驗證通常會分為：

- `Stage 1`：文件審查，確認 ISMS 文件、範圍、風險評鑑、政策、程序等是否具備基本條件。
- `Stage 2`：實地審查，確認制度是否實際運作，並查核紀錄、訪談人員、檢視控制措施執行情形。

證書週期通常 3 年，期間仍有定期監督審查。

**下列何者「不」是 ISO/IEC 27001:2022 轉版變化的重點？**  
(A) 管理系統與 ISO 結構的一致性  
(B) 簡化標題  
(C) 新增控制項目  
***(D) 驗證範圍調整***  

關鍵判斷：  
ISO/IEC 27001:2022 相較 2013 版，主要變化包含：

- 管理系統條文更貼近 ISO harmonized structure。
- 條文與標題用語有所簡化與調整。
- Annex A 控制項目依 ISO/IEC 27002:2022 重整，控制項目由 114 項調整為 93 項，並新增若干控制項。

但「驗證範圍調整」不是標準本身轉版的必然重點。組織可以在轉版時檢討驗證範圍，但那是組織個別管理決策，不是 ISO 27001:2022 改版主軸。

**風險是指不確定性對目標的影響，或是損失的不確定性。下列何項「非」不確定性所指的三種情況之一？**  
(A) 不確定事件是否發生  
(B) 不確定何時發生  
***(C) 不確定發生的地點***  
(D) 不確定造成的嚴重性  

關鍵判斷：  
風險中的不確定性，通常可理解為：

- 事件是否會發生
- 事件何時會發生
- 事件若發生，會造成多大影響或損害

因此「不確定發生的地點」不是本題所稱三種情況之一。

**採用 FIDO2 作為身分認證是目前的趨勢，關於 FIDO2 的敘述下列何者有誤？**  
***(A) 須加入 FIDO 聯盟認證***  
(B) 可以應用在企業內部作為身分認證  
(C) 可以應用在網際網路之金融應用  
(D) 是屬於零信任架構的一環  

關鍵判斷：  
`FIDO2` 是以公開金鑰密碼學為基礎的現代身分驗證機制，包含：

- `WebAuthn`：瀏覽器／網站端 API，W3C 標準
- `CTAP`：Client to Authenticator Protocol，用於用戶端與驗證器之間通訊

採用 FIDO2 進行身分驗證，不代表組織一定要加入 FIDO Alliance 或取得聯盟會員身分。  

判斷規則：  
- FIDO2 = WebAuthn + CTAP
- 公鑰驗證，私鑰留裝置端
- 可用於企業、金融、零信任身分驗證
- 採用 FIDO2 ≠ 必須加入 FIDO 聯盟

**政府零信任網路架構係參考 NIST 零信任架構，同時結合向上集中之防護需求，採取資源門戶部署（Resource PortalBased Deployment）方式，下列何項「不」屬於該核心機制？**  
(A) 身分鑑別 ***(B) 系統識別*** (C) 設備鑑別 (D) 信任推斷  

關鍵判斷：  
政府零信任網路架構採資源門戶部署方式，核心機制為：

- `身分鑑別`：確認使用者身分。
- `設備鑑別`：確認使用設備與設備健康狀態。
- `信任推斷`：依使用者、設備、來源 IP、登入時間、設備健康等情境資料判斷是否允許存取。

因此「系統識別」不是三大核心機制之一。

**在檢查一個利用第三方服務的電子資料交換系統（EDI）應用情況時，資訊安全人員應該確認和證實哪些項目？**  
**1. 確認加密金鑰符合使用者需求**  
**2. 證實服務供應商只使用了 PSDN（Public Switched Data Network）**  
**3. 確認是否已經對服務供應商的營運情況進行了獨立檢查**  
**4. 證實服務供應商的合約包了必要的條款，例如稽核查核的權力**  
(A) 1、3 (B) 1、4 (C) 2、3 ***(D) 3、4***  

關鍵判斷：  
第三方 EDI 服務涉及外部服務供應商，因此資訊安全人員應確認：

- 是否已經對服務供應商的營運情況進行獨立檢查  
- 服務供應商合約是否包含必要條款，例如稽核查核權利  

這兩項能確認供應商控制措施是否可信，以及組織是否保留必要監督權限。

**「生日攻擊法」（Birthday Attack）主要針對下列何種加密演算法進行攻擊？**  
***(A) MD5*** (B) Rijndael (C) 3DES (D) RSA  

關鍵判斷：  
- `Birthday Attack` 利用生日悖論的概念，攻擊雜湊函數的碰撞抗性。
- 它關注的是：是否能找到兩個不同輸入，但產生相同 hash value，也就是 collision
- MD5 是雜湊函數，且已知碰撞攻擊風險高，因此最符合本題。

判斷規則：  
- 生日攻擊 → Hash collision → MD5 / SHA-1 這類雜湊要小心

**關於組織進行營運衝擊分析所需包含之內容敘述，下列何者錯誤？**  
(A) 辨識關鍵業務相關之活動  
(B) 辨識各活動中斷時，會對機關所造成的衝擊  
***(C) 確認各活動之最小可容忍中斷時間（Minimum Tolerable Period of Disruption, MTPD）***   
(D) 根據各活動復原之優先順序加以分類，藉以鑑別機關之關鍵活動  

關鍵判斷：  
`MTPD` 是：`Maximum Tolerable Period of Disruption`，也就是：`最大可容忍中斷時間`
