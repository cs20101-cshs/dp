## 資訊安全技術概論
有關 BadUSB 的敘述，下列何者正確？  
(A) BadUSB 是一種惡意軟體，僅會感染 USB 隨身碟內的檔案資料  
(B) BadUSB 攻擊僅限於 Windows 作業系統，不會影響 macOS 或 Linux  
(C) BadUSB 攻擊無法被一般的防毒軟體偵測並阻擋  
***(D) BadUSB 利用修改 USB 裝置的韌體，使其在電腦上被辨識為其他硬體裝置（如鍵盤）***  

關鍵判斷：  
`BadUSB` 是利用 USB 裝置韌體或控制器可被修改的特性，讓 USB 裝置偽裝成其他類型的裝置，例如：

- 鍵盤
- 滑鼠
- 網路卡
- 大容量儲存裝置

常見攻擊方式是讓 USB 被辨識為鍵盤後，自動輸入指令，或偽裝成網路裝置改變流量路徑。  

判斷規則：  
- `BadUSB` → 改 USB 韌體／裝置行為。
- 偽裝成鍵盤等裝置 → 自動輸入惡意指令。
- 不限 Windows。
- 不是單純檔案感染。

**在 SSL/TLS 通訊協定中，客戶端與伺服器會在握手階段（Handshake）交換金鑰與協定版本資訊。下列何項最能説明為什麼在握手過程中要同時交換「隨機數（Random Values）」與「Session ID」等資訊？**  
(A) 為了減少握手所需的封包傳輸次數，提升傳輸速率  
***(B) 為了確保金鑰獲取（Key Derivation）具有多樣性與不可預測性***  
(C) 為了讓伺服器能夠在握手過程中壓縮客戶端的傳輸內容  
(D) 為了讓客戶端在握手後能夠複用過去的憑證（Certificate）而不再驗證  

關鍵判斷：  
SSL/TLS 握手時，客戶端與伺服器會交換：

- `Client Random`
- `Server Random`
- 協定版本
- Cipher Suite
- Session ID / Session Ticket 等資訊

其中 `Client Random` 與 `Server Random` 會參與 session key 的推導，使每次連線產生不同的加密金鑰，避免金鑰可預測或重複。  

判斷規則：  
- TLS Random Values → 參與金鑰推導，確保每次 session key 不同
- Session ID → session resumption / 辨識既有 session

**在資料完整性保護上，當使用雜湊函數（例如 SHA-256）進行訊息驗證時，下列哪種方法最能防止「長度延展攻擊（Length Extension Attack）」？**  
(A) 直接使用 SHA-256 雜湊結果作為簽章  
(B) 使用常見的 MD5 雜湊函數來保護資料  
(C) 在雜湊前對訊息加上高熵的隨機鹽值（Salt），並在驗證時同樣加入該鹽值  
***(D) 使用 HMAC（Hashed Message Authentication Code）機制進行驗證***  

關鍵判斷：  
如果直接把：`SHA-256(secret || message)` 當成訊息驗證碼，攻擊者在不知道 secret 的情況下，仍可能利用長度延展攻擊，推算出：

`SHA-256(secret || message || padding || attacker_data)`

也就是可以在原訊息後面接資料，並偽造出看似有效的雜湊值。這就是雜湊函數被拿來當 MAC 的經典自殺姿勢。

`HMAC` 的設計會使用內外兩層雜湊與 key mixing：`HMAC(key, message)`，因此可以避免單純 Merkle–Damgård hash 被長度延展攻擊利用。

判斷規則：  
- Length Extension Attack → 不要直接用 hash(secret || message)
- 防法 → HMAC
- Salt ≠ HMAC
- Hash ≠ 簽章

**下列哪一項是幽靈攻擊（Spectre）利用現代處理器的特性來竊取敏感資訊？**  
(A) 記憶體交換（Memory Swapping）  
***(B) 分支預測（Branch Prediction）和推測執行（Speculative Execution）***  
(C) 虛擬記憶體（Virtual Memory）  
(D) 多重執行緒處理（Multithreading）  

關鍵判斷：  
`Spectre` 是利用現代處理器的：

- 分支預測
- 推測執行
- 快取側通道

讓攻擊者誘導 CPU 在推測路徑中存取不該存取的資料，再透過快取時間差推測敏感資訊。

判斷規則：  
- `Spectre` → Branch Prediction / Speculative Execution
- `Meltdown` → 利用亂序執行與權限檢查時序問題，讀取受保護記憶體

**下列何項「不」屬於程式開發過程中的安全檢測項目？**  
(A) 動態應用系統安全測試 DAST（Dynamic Application Security Testing）  
***(B) 系統冒煙測試（Smoke Testing）***  
(C) 原始碼測試（Source Code Testing）  
(D) 白箱測試（White Box Testing）  

關鍵判斷：  
`Smoke Testing` 是一般軟體測試裡的基本可用性檢查，通常用來確認系統主要功能是否能跑、版本是否能進一步測試，它不是專門的安全檢測方法。

判斷規則：  
- `DAST` → Dynamic Application Security Testing，動態應用系統安全測試，屬於安全檢測。
- `Smoke Testing` → 冒煙測試，確認系統基本功能是否正常，不屬於安全檢測主項目。
- `Source Code Testing` → 原始碼測試／檢查，可對應靜態分析、原始碼安全檢測，屬於安全檢測範圍。
- `White Box Testing` → 白箱測試，可用於安全檢測，因為測試者知道內部程式邏輯、架構與原始碼。

**為了躲避安全研究人員的分析與惡意軟體偵測機制，某些惡意程式會嘗試偵測自己是否執行於沙盒環境（Sandbox）內，並在偵測到時隱藏或延遲執行惡意行為。下列何種方法「不」是惡意程式常見的沙盒偵測技術？**  
(A) 檢查系統運行時間是否極短，以判斷是否為剛啟動的虛擬環境  
(B) 嘗試存取特定的虛擬機或沙箱環境特徵，如硬體資訊  
(C) 檢查是否有鍵盤或滑鼠輸入活動，以確認系統是否由真人操作  
***(D) 自動將自身程式碼重新加密以防止靜態分析***  

關鍵判斷：  
惡意程式的沙盒偵測重點是判斷：

- 我是不是跑在虛擬機？
- 我是不是跑在分析沙盒？
- 這台機器是不是太乾淨、太新、太不像真人使用？
- 有沒有滑鼠鍵盤活動？
- 硬體資訊、驅動程式、程序、檔案、registry 有沒有 VM / sandbox 特徵？

「重新加密自身程式碼」則是用來防止靜態分析、增加逆向難度，屬於 packer / obfuscation / anti-static-analysis 技術，不是沙盒偵測本身。  

判斷規則：  
- Sandbox detection = 判斷環境是不是沙盒
- Code encryption / packing = 防靜態分析，不是偵測沙盒

**下列何項「不」是常用來觀察駭客攻擊行為的模型（Models）？**  
(A) MITRE ATT&CK (B) MITRE D3FEND (C) MITRE ENGAGE ***(D) MITRE DEFEND***  

關鍵判斷：  
常見 MITRE 相關模型／框架：

- `MITRE ATT&CK`  
  用來描述攻擊者的戰術、技術與程序，常用於觀察與分析攻擊行為。
- `MITRE D3FEND`  
  防禦技術知識圖譜，對應攻擊技術與防禦方法。
- `MITRE ENGAGE`  
  與主動防禦、欺敵、誘捕、對抗互動有關，用來理解與規劃 adversary engagement。

**某公司每次完整備份的資料量約略少於 800GB，公司的顧問建議在備份媒介的選擇上，每一個單一的儲存媒介，需可以完整存放其一次的備份資料，請問下列哪種儲存媒介較為合適？**  
(A) BD-R (B) LTO-2 ***(C) LTO-4*** (D) DAT320  

關鍵判斷：  
常見備份媒體容量約略如下：

- `BD-R`：常見 25GB，雙層 50GB，遠低於 800GB
- `LTO-2`：原生 200GB，壓縮約 400GB
- `LTO-4`：原生 800GB，壓縮約 1.6TB
- `DAT320`：原生 160GB，壓縮約 320GB

**某網路管理員收到要求，需要開放在 DMZ 區的伺服器 A，讓其可以透過 RFC 5424 的協定寫入相關的日誌記錄到 DMZ 區以外的區域中的伺服器 B，請問以下列何項設定較為合適？**  
(A) 允許所有伺服器到伺服器 B 的連線  
(B) 允許所有來自伺服器 A 到伺服器 B 的連線  
(C) 允許伺服器 A 至伺服器 B 中 TDP Port 514 的連線  
***(D) 允許伺服器 A 至伺服器 B 中 UDP Port 514 的連線***  

關鍵判斷：  
RFC 5424 是 Syslog 訊息格式相關標準。Syslog 傳統常見使用 `UDP 514` 傳送日誌。

題目情境是：
- DMZ 區伺服器 A
- 要寫入日誌到 DMZ 外伺服器 B
- 因此防火牆規則應盡量精準，只允許必要來源、目的與服務。

所以應允許：  
`Server A → Server B / UDP 514`

**在台灣，行動裝置的 App 要送「行動應用 App 基本資安檢測實驗室」檢測時，如果有金流及會員資料的，是屬於下列何項等級？**  
(A) L1 (B) L2 ***(C) L3*** (D) L4  

關鍵判斷：  
行動應用 App 基本資安檢測基準：  
- `L1`：無須使用者身分鑑別。
- `L2`：須使用者身分鑑別。
- `L3`：含交易行為／金流交易功能。
- `F`：安全性需求較高之加測類別。

判斷規則：  
- 無登入 → L1
- 有登入 → L2
- 有交易／金流 → L3
- 高安全需求 → F 加測

## 資訊安全管理概論
**因應客戶的要求，貴公司擬導入個人資料保護的相關標準，因兩個月前您才幫公司完成 ISO 27001:2022 轉版驗證；為達最佳效益，試問您會建議公司導入下列何項認證標準？**  
(A) ISO 10012 ***(B) ISO 27701*** (C) ISO 27017 (D) ISO 27018  

關鍵判斷：  
`ISO/IEC 27701` 是隱私資訊管理系統（Privacy Information Management System, PIMS）標準，用於個人資料／隱私管理。

判斷規則：  
- 量測管理 → ISO 10012
- 個資 / 隱私管理系統 → ISO 27701
- 雲端安全 → ISO 27017
- 公有雲 PII 保護 → ISO 27018

---
**關於 ISO 27001:2022 標準中，要求文件化資訊的項目，下列哪一項錯誤？**  
(A) 資訊安全政策  
***(B) 最高管理階層的領導作為***  
(C) 資訊安全管理系統（ISMS）範圍  
(D) 資訊安全目標  

關鍵判斷：  
ISO 27001:2022 中常見要求文件化資訊的項目包含：

- ISMS 範圍
- 資訊安全政策
- 資訊安全目標
- 風險評鑑與風險處理相關資訊
- 適用性聲明 SoA
- 稽核、管理審查、矯正措施等紀錄

但「最高管理階層的領導作為」屬於高階管理者應展現的領導與承諾，不是列為文件化資訊的項目。

---
**在 ISO 27001 的遠距工作管理規範下，若一個組織決定完全禁止遠距工作，並在其聲明中將該控制措施標註為「不適用」，下列何者描述最能與該聲明相符？**  
(A) 確保現階段組織內無員工進行遠距工作，並在文件中說明該控制項不適用  
(B) 確認組織未來可能的業務需求和風險變化，以便根據需要靈活調整遠距工作政策  
***(C) 確認並記錄在所有情況下（包括應急或特殊情況）都禁止遠距工作，並提供相應的證據***  
(D) 定期審查和更新遠距工作政策，即使目前遠距工作被禁止，仍需為未來需求保留調整空間  

關鍵判斷：  
ISO 27001 的控制項若標示為「不適用」，不是口頭說「我們目前沒有」就好，必須能說明：
- 為什麼該控制項不適用
- 組織是否真的沒有該風險情境
- 是否在所有相關情況下都排除該活動
- 是否有政策、程序或紀錄可支持這個判斷

**數位憑證（Digital Certificate）是在網際網路上識別人員和資源的電子檔案，能夠讓兩個實體之間能夠進行安全、機密的通訊。請問下列何項「不」是數位憑證的正確描述？**  
(A) 使用在伺服器上的數位憑證又稱為「伺服器憑證」、HTTPS/SSL 憑證，SSL 憑證是在網頁伺服器（主機）與網頁瀏覽器（客戶端）之間建立一個密碼連結的標準規範  
***(B) 一般網頁用的 SSL 憑證檔案中包含公鑰資訊、擁有者身分資訊、數位憑證發行者、憑證有效到期日，但是不會出現發行憑證日期***  
(C) 大部分的瀏覽器會在網址列裡顯示一個鎖的圖示或是其他識別方式，如果要了解 SSL 憑證的細節，只要點擊鎖的圖示即可  
(D) 憑證用途非常廣泛，像是自然人憑證、工商憑證、健保卡等都是屬於數位憑證的用途，幾乎身分與機構識別都有利用到此技術  

關鍵判斷：  
數位憑證通常包含：
- 主體／擁有者資訊
- 公鑰資訊
- 憑證發行者
- 憑證序號
- 簽章演算法
- 憑證有效期間
  - Not Before
  - Not After
- 發行者的數位簽章

憑證中會有有效起始時間與有效結束時間，因此不能說「不會出現發行憑證日期」或相關時間資訊。

判斷規則：  
- 憑證 = 身分資訊 + 公鑰 + CA 簽章 + 有效期間
- 憑證含公鑰，不含私鑰
- X.509 有 Not Before / Not After

**美國、歐盟及我國皆有針對該國家的企業組織、關鍵基礎設施等機構，規範於發現或發生資安事件時應辦理的相關事項，其中有關重大事件應通報主管機關的時效規定，請問，下列何者的時間為「最短」？**  
(A) 我國《資通安全事件通報及應變辦法》 
***(B) 我國《證券期貨市場資通安全事件通報應變作業注意事項》***  
(C) 歐盟《The Digital Operational Resilience Act》  
(D) 美國《Cyber Incident Reporting for Critical Infrastructure Act》  

關鍵判斷：  
本題比較重大資安事件通報主管機關的時效，最短者為證券期貨市場規範：

- 證券期貨業者：知悉事件後 `30 分鐘內` 初步通報。
- 我國資通安全事件通報及應變辦法：知悉事件後 `1 小時內` 通報。
- EU DORA：重大 ICT 事件初步通知為數小時至 24 小時量級。
- 美國 CIRCIA：covered cyber incident 通常為 `72 小時內`。
