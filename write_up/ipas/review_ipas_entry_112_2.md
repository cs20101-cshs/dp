# iPAS 資安工程師初級考古題 112-2 錯題檢討

## 資訊安全技術概論
**在我們使用網路掃描工具 nmap 進行網路探勘時，nmap 通常會先透過 ping 來確認目標伺服器是否可以連線，如果目標伺服器關閉 ping 回應功能，可能會導致 nmap 誤判目標伺服器並未上線。請問下列何項 nmap 指令可以跳過 ping 功能直接開始掃 port？**  
A) -sL ***(B) -Pn*** (C) -p- (D) -sn  

關鍵判斷：  
- `nmap` 預設會先做 host discovery，常見方式包含 ping / probe，確認目標是否存活。如果目標主機關閉 ICMP echo reply 或封鎖探測封包，nmap 可能誤判主機未上線。
- 使用 `-Pn` 可以跳過 host discovery，直接把目標視為 online，進行 port scan。  

判斷規則：  
- `-sL` → 只列清單，不掃描
- `-Pn` → 不 ping，直接掃（no ping）
-  `-p-` → 掃全部 port
- `-sn` → 只 ping，不掃 port（no port scan）

**網站應用程式伺服器允許顯示錯誤訊息給使用者，如：Stack traces，將可能導致機敏資訊外洩或底層元件的弱點被發現，關於此應用程式缺陷的描述，下列何者較正確？**  
(A) 跨站請求偽造（Cross-Site Request Forgery）   
***(B) 安全設定錯誤（Security Misconfiguration）***  
(C) 認證和驗證失效（Identification and Authentication
Failures）  
(D) 伺服器端請求偽造（Server-Side Request Forgery）  

關鍵判斷：  
網站應用程式若允許使用者看到詳細錯誤訊息，例如：  
- Stack traces
- 程式路徑
- 資料庫錯誤
- 框架或伺服器版本
- 內部元件資訊

可能導致機敏資訊外洩，或讓攻擊者知道底層元件與弱點位置。這屬於**安全設定**錯誤。    

判斷規則：  
- `使用者被騙去送請求` → `CSRF`
- `Stack trace`、`debug message`、`詳細錯誤訊息外洩` → `Security Misconfiguration`
- `登入、密碼、session、認證機制` → `Identification and Authentication Failures`
- `伺服器被誘導去打 URL / 內網服務` → `SSRF`

**關於應用程式軟體與資料完整性的敘述，下列何者錯誤？**  
(A) 使用數位簽章（Digital Signature）來驗證軟體未被竄改  
(B) 僅於信任的儲存庫（Repository）取得應用程式套件  
(C) 整合測試流程（CI/CD）應適當隔離與存取控制以確保安全  
***(D) 源碼掃描可有效檢查是否遭植入惡意程式碼（如：後門程式）***  

關鍵判斷：  
- 「源碼掃描」在本題指自動化源碼掃描工具，主要用於檢查常見弱點與不安全寫法。  
- 若是資深工程師進行人工 code review，確實可能發現可疑後門或惡意邏輯；但自動化源碼掃描不能保證有效檢查是否遭植入惡意程式碼。

判斷規則：  
- `數位簽章` → 驗證完整性與來源。
- `可信 repository` → 降低套件供應鏈風險。
- `CI/CD 隔離與權限控管` → 防止建置流程被污染。
- `源碼掃描` → 自動化工具找弱點，不等於可靠抓後門。

**從 Microsoft 安全性開發週期（Secure Development Lifecycle, SDL）的觀點，靜態程式分析（Static Analysis）主要是下列何項階段的考量？**  
(A) 需求階段（Requirements）  
(B) 設計階段（Design）  
***(C) 實作階段（Implementation）***  
(D) 分析階段（Analysis）  

關鍵判斷：  
- 靜態程式分析（Static Analysis）是在不執行程式的情況下檢查原始碼或編譯後程式，以找出常見弱點、不安全寫法或潛在錯誤。
- 從 Microsoft SDL 觀點來看，Static Analysis 主要是在程式碼撰寫期間進行，因此屬於 `Implementation` 實作階段。

判斷規則：  
- `安全需求` → Requirements
- `威脅建模 / Attack Surface Reduction` → Design
- `Static Analysis / 安全程式碼 / code review` → Implementation
- `最終安全檢查 / 發佈核准` → Release
- **沒有 Analysis 階段**

**關於惡意程式防護與弱點管理的敘述，下列何者錯誤？**  
(A) IOC（Indicators of Compromise）稱作入侵威脅指標  
(B) TTPs（Tactics, Techniques and Procedures）是指戰術、技術與程序，被應用在網路安全  
***(C) IOA（Indicators of attack）攻擊指標，會記錄攻擊來源、hash 值或 URL***  
(D) Zero Trust Security 也是一種惡意程式防護與弱點管理手段  

關鍵判斷：  
`IOC` 和 `IOA` 的差別：  
- `IOC`：Indicators of Compromise，偏「已遭入侵或可疑活動留下的跡證」，常見包含：惡意 IP、Domain、URL、檔案 hash、惡意檔案名稱、Registry key、可疑路徑
- `IOA`：Indicators of Attack，偏「攻擊行為或攻擊模式」，重點是攻擊者正在做什麼，例如：掃描、暴力破解、憑證嘗試、權限提升、橫向移動、可疑 PowerShell 行為、異常程序鏈  

判斷規則：  
- `IOC` → 看「留下什麼跡證」
- `IOA` → 看「正在做什麼攻擊行為」
- `hash / URL / IP / domain` → 多半是 `IOC`
- `TTPs` → 攻擊者的戰術、技術、程序

**從 Microsoft 安全性開發週期（Secure Development Life, SDL）的觀點，縮小攻擊面向（Attack Surface Reduction），主要是下列何項階段的考量？**  
(A) 需求階段（Requirements）  
***(B) 設計階段（Design）***  
(C) 實作階段（Implementation）  
(D) 發佈階段（Release）  

關鍵判斷：  
- 縮小攻擊面向（Attack Surface Reduction）是在設計系統時，盡量減少可被攻擊者利用的入口、服務、功能、介面與權限暴露。通常屬於安全**設計階段**的考量，而不是等到實作或發佈才補救。  

判斷規則：  
- `安全需求`、`隱私需求`、`合規要求` → Requirements
- `威脅建模`、`攻擊面縮小`、`安全架構` → Design
- `安全程式碼`、`靜態分析`、`程式碼檢查` → Implementation
- `最終安全檢查`、`事件應變計畫`、`發佈核准` → Release

**通用漏洞評分系統（Common Vulnerability Scoring System, CVSS）定義了數個指標組用來評定電腦系統安全的嚴重性及風險程度，請問有關CVSS 3.1評分指標組之敘述，下列何者正確？**  
(A) 強制的基本指標組、強制的時間指標組以及可選擇的環境指標組  
(B) 強制的基本指標組、強制的環境指標組以及可選擇的時間指標組  
(C) 可選擇的基本指標組、強制的時間指標組以及可選擇的環境指標組  
***(D) 強制的基本指標組、可選擇的環境指標組以及可選擇的時間指標組***  

關鍵判斷：  
CVSS 3.1 評分指標組分為三類：
- `Base Metrics`：基本指標組  
  描述漏洞本身固有特性，例如攻擊向量、攻擊複雜度、所需權限、使用者互動、影響範圍、機密性／完整性／可用性影響。  
  這組是必要項目。  
- `Temporal Metrics`：時間指標組  
  描述隨時間改變的因素，例如利用程式成熟度、修補狀態、報告可信度。  
  這組是選填。
- `Environmental Metrics`：環境指標組  
  依照特定組織環境調整風險，例如資產重要性、環境中的 CIA 需求等。  
  這組是選填。

判斷規則：  
- `Base` → 強制
- `Temporal` → 選填
- `Environmental` → 選填

**企業為了檢查自己是否會被已知的弱點攻擊，通常會定期安排弱點掃描，以便檢查是否有應用程式或作業系統的漏洞出現在企業的環境中。下列何項「不」是常見的弱點掃描工具？**  
(A) OSV-Scanner (B) Nessus (C) OpenVAS ***(D) Masscan***  

關鍵判斷：  
弱點掃描工具通常會檢查系統、套件、服務或應用程式是否存在已知漏洞。  
- `Nessus`：商業弱點掃描工具，經典老牌。
- `OpenVAS`：開源弱點掃描工具。
- `OSV-Scanner`：用來掃描專案相依套件是否存在 OSV 資料庫中的已知漏洞。

`Masscan` 則主要是高速 port scanner，用來大量掃描 IP 與 port 開放狀態，不是典型弱點掃描工具。  

判斷規則：  
- `Nessus`、`OpenVAS` → 弱點掃描
- `OSV-Scanner` → 套件／相依性漏洞掃描
- `Masscan` → 高速 port 掃描

**關於 Linux 系統日誌資料的敘述，下列何者正確？**  
(A) /var/log/lastlog 記錄系統中所有用戶最後一次的登錄時間。可直接用 vi 查閱，但需搭配使用 lastlog 命令進行  
(B) /var/log/cron 記錄與系統設定任務相關的日誌  
(C) /var/log/dmesg 記錄系統關機時，內核自我檢查資訊，也可以使用 dmesg 命令直接查看內核自檢資訊  
(D) /var/log/secure 記錄郵件資訊的日誌  

選項解析：  
- `/var/log/lastlog` 記錄所有使用者最後一次登入時間  
  它是二進位格式，不是拿 `vi` 直接看；通常用 `lastlog` 指令查看。
- `/var/log/cron` 記錄與系統排程任務相關的日誌  
  正確。cron job 執行情況會記在這裡，依發行版設定也可能進 syslog。
- `/var/log/dmesg` 主要看 kernel ring buffer，常見是開機與核心訊息，不是「關機時」的自我檢查。
- `/var/log/secure` 常見於 Red Hat 系，記錄認證、安全相關資訊，例如 SSH、sudo、su。郵件通常會看 `/var/log/maillog` 或 `/var/log/mail.log`。

## 資訊安全管理概論
**關於資訊安全管理系統的敘述，下列何者錯誤？**  
(A) ISO 27001：2022 是目前最新版本  
***(B) 建立資訊安全管理系統必須以 ISO 27001 為架構***  
(C) 機密性、可用性、完整性是 ISO 27001 最主要的資安 3 要素  
(D) ISO 27001資訊安全管理系統是一個持續改善的架構  

關鍵判斷：  
- `ISO/IEC 27001:2022` 是資訊安全管理系統（ISMS）的國際標準，定義 ISMS 應符合的要求，也提供組織建立、實施、維護與持續改善 ISMS 的架構。ISO 官方頁面目前列示 ISO/IEC 27001:2022 為第 3 版，2022-10 發布，狀態為 Published。

- 但建立 ISMS 不必然「必須」以 ISO 27001 為架構。ISO 27001 是常用且可認證的標準，不是唯一可能的管理架構。

**若要驗證資訊安全管理系統，且能有效達成稽核作業，如稽核方案之管理、內部稽核之執行等，可參考下列哪一項標準？**  
(A) ISO 27002 ***(B) ISO 27007*** (C) ISO 27017 (D) ISO 27037  

判斷規則：  
- `ISO 27001` → ISMS 要求／可驗證標準
- `ISO 27002` → 控制措施實作指引
- `ISO 27007` → ISMS 稽核指引
- `ISO 27017` → 雲端安全
- `ISO 27037` → 數位證據處理

(A) 業務涉及全國性民眾服務或跨特定非公務機關共用性資通系統之維運，其資通安全責任等級為 A 級

***(A) 蒐集：指以特定方式取得個人資料***

**依據營業秘密法的規定，下列何者正確？**  
(A) 工程師於職務上研究或開發之營業秘密，歸工程師所有  
(B) 營業秘密不得讓與他人或與他人共有  
***(C) 營業秘密不得為質權及強制執行之標的***  
(D) 營業秘密指方法、技術、製程、配方，程式則不屬營業秘密，應屬著作權  

關鍵判斷：  
《營業秘密法》條文：  
- 第 2 條：  
    本法所稱營業秘密，係指方法、技術、製程、配方、程式、設計或其他可用於生產、銷售或經營之資訊，而符合左列要件者：
    一、非一般涉及該類資訊之人所知者。
    二、因其秘密性而具有實際或潛在之經濟價值者。
    三、所有人已採取合理之保密措施者。  
- 第 3 條第 1 項：  
    受雇人於**職務上**研究或開發之營業秘密，歸**雇用人**所有。但契約另有約定者，從其約定。

    出資聘請他人從事研究或開發之營業秘密，其營業秘密之歸屬依契約之約定；契約未約定者，歸受聘人所有。但出資人得於業務上使用其營業秘密。  
- 第 6 條第 1 項：  
    營業秘密得全部或部分讓與他人或與他人共有。
- 第 8 條：  
    營業秘密不得為質權及強制執行之標的。

**關於 HTTPS 協議中所使用數位憑證（Digital Certification）的敘述，下列何項錯誤？**  
(A) 其為 X.509 電腦網路標準，並使用 X.500 資料模組定義屬性  
(B) 用於驗證所代表之實體（Entity），例如：主機、組織或個人  
(C) 可透過憑證確認其發行者的數位簽章（Digital Signature）  
***(D) 每張憑證僅能用於驗證一個網站（Website）***  

關鍵判斷：  
- HTTPS 使用的數位憑證通常是 `X.509` 憑證，用來證明憑證所代表的實體身分，例如主機、組織或個人。瀏覽器會檢查憑證鏈與憑證發行者的數位簽章，以確認憑證是否可信。
- 但一張憑證不一定只能驗證一個網站。透過 `Subject Alternative Name（SAN）` 或萬用字元憑證，可以讓同一張憑證涵蓋多個網域名稱。

判斷規則：  
- `X.509` → HTTPS 常用數位憑證格式。
- `憑證` → 驗證實體身分與公鑰。
- `CA 簽章` → 建立信任鏈。
- `SAN / Wildcard` → 一張憑證可涵蓋多個網站。

**在組織遇到資安事件時，可能需要尋找適宜的替代場地，請問關於替代場地的敘述，下列何者較正確？**  
(A) 冷備援站點（Cold site）：場地和設備皆有，於事件發生時依需求執行回復工作  
(B) 暖備援站點（Warm site）：異地備援端備有同樣之系統，事件發生時在可接受的時間內恢復啟用，待資料載入後即可投入營運  
(C) 熱備援站點（Hot site）：有提供基本設備，但仍須自行安裝系統  
***(D) 行動備援站點（Mobile hot site）：在移動車輛中安裝設備與系統，當事件發生時，可迅速移動至適當地點啟用***  

關鍵判斷：  
替代場地大致可分：  
- `Cold site`：只有場地與基本設施，通常沒有完整設備或系統，啟用時間最長。
- `Warm site`：已有部分設備與系統，資料可能需要載入或同步到某一程度，啟用時間中等。
- `Hot site`：設備、系統與資料都接近可立即使用，啟用時間最短。
- `Mobile site / Mobile hot site`：以車輛或可移動設備提供備援環境，可移動到指定地點啟用。

判斷規則：  
- `Cold site` → 有場地，最少設備，恢復最慢。
- `Warm site` → 有部分設備／系統，資料需補齊，恢復中等。
- `Hot site` → 幾乎可立即接手，恢復最快。
- `Mobile site` → 可移動備援環境。
